# Requirements

<TOCInline toc={toc}/>

## Software Artifacts

#### R-SArt-001: Documentation

- The system, including its architecture, must be well documented.
- Design decisions must be documented using Architectural Design Records (ADRs).

#### R-SArt-002: Clean code

- The code must be reviewed for quality, especially clarity and brevity.

#### R-SArt-003: Clean and simple deployment

- A *one-click deployment* is not strictly required, but the deployment process must be as simple and clean as possible.
- The README file referring to the deployment must be complete, well-structured, informative, easy to read and to understand, and always kept up to date.
- A new administrator without prior knowledge about the system should be able to deploy the system within 2 hours by following the instructions in the README file.

#### R-SArt-005: Chaos testing

- The system must be tested using chaos testing.
- Depending on the selected chaos testing toolkit, the system must support the dynamic injection of faults and network conditions (e.g., packet loss, latency) to simulate failure scenarios.
- Regarding the targeted non-functional requirement at test -- reliability --: Either the toolkit simulates unreliability or the system must support different configurations of reliability itself.

## Product

:::warning
### Domain

#### R-D-001: The domain must be specific and self-explanatory

- The domain must be a specific one.
- Researchs must be able to understand the domain intuitively.
- The domain must not be an obstacle under any circumstances. As long as the domain is _self-explanatory_, it doesn't matter which domain is shown.
- The domain must not require special knowledge about it in order to be able to use the system.

#### R-D-002: The domain should not be part of the _technology world_

- The domain should be clearly separated from any technology domain to avoid confusion regarding wordings and terms.
- There should be no clashes regarding the language of the domain and the language of the system's architecture: no common wordings.
- e.g., words like service, pipeline, container (Docker) should be avoided in the domain language

:::

### Usability Requirements

#### R-U-003: Configurations

- The system exposes exactly one configuration file in a central location. This file must be independent of the number of microservices and their technology stacks.

#### R-U-008: Observability

- The administrator must be able to observe and understand what the system is doing. To achieve this, fine-grained logging and monitoring must be in place.
- The current utilization (load) of the individual microservices must be reported via a OpenTelemetry monitoring API.

#### R-U-014: Demo mode

- There should be a kind of demo mode so that a new user can try out the system quickly and easily: We need to provide test data with which the system can be directly _loaded_.

#### R-U-018: Usage costs

- The system should measure and report the costs of running and scaling the system.
- As an administrator, I want to make decisions based on the costs of the system's usage.

#### R-U-019: JMeter testing

- The system must be load tested using JMeter.
- The documentation should guide performance engineers in setting up test scenarios accurately within JMeter.
- The documentation should define clear API contracts or specifications (such as Swagger/OpenAPI documentation) for the System's endpoints, request formats, and responses. This helps in accurately configuring JMeter's HTTP Request samplers and assertions.

### Non-functional Requirements

#### R-NFR-002: Elasticity / Auto scaling

- The system must be _elastic_.
- The system must be able to scale automatically.

#### R-NFR-003: Resilience

- The system should be resilient.
- The systems' microservices must not fail suddenly without good reason. They should especially continue to operate when other microservices fail.

#### R-NFR-004: Maintainability

- In order to improve maintainability, things should be kept as simple as possible.
- Instead of trying to support every possible deployment, only one deployment should be supported but that one should be supported very well.

### Configurations

#### R-Conf-004: Scaling policies

- The system should offer different alternatives for scaling.
- The system should allow the user to configure the scaling techniques in place.
- For each scaling strategy, the system should inform about the costs of the chosen scaling strategy.

#### R-Conf-006: Configurations scope

- The system should allow for configurations on the following levels:
  - System
  - Microservice
  - Database (Management System)

## Design

### Architecture

#### R-A-001: Extendability

- The System must be expandable: Additional microservices can be added without further ado.

#### R-A-002: Scale out scenario for databases

- The system should account for a scale out scenario regarding databases.

#### R-A-003: Distributed Transactions

- The system should support distributed transactions, ensuring the data integrity across multiple microservices and databases.

#### R-A-005: Loose Coupling

- Microservices must be only loosely coupled. All communication must be via the other microservices' APIs.
- A domain-driven declaration of microservice responsibilities must be at hand.

#### R-A-008: Distributed Tracing

- The system must have distributed tracing available.
