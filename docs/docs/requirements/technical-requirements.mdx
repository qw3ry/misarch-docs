# Technical Requirements

## 1. Administration

### Deployment

The deployment process should be as simple as possible

- **TR-1.1**: The README file referring to the deployment must be complete, well-structured, informative, easy to read and to understand, and always kept up to date.
- **TR-1.2**: A new administrator without prior knowledge about the system should be able to deploy the system within 2 hours by following the instructions in the README file.
- **TR-1.3**: The system shall have a demo mode that supplies dummy data, so the user can try the system without adding data.

### Scalability

The system must be able to handle different loads automatically.

- **TR-1.4**: The system must be _elastic_ and support automatic scaling.
- **TR-1.5**: The system shall allow different scaling strategies that can be configured.

### Observability

The administrator must be able to observe and understand what the system is doing.

- **TR-1.6**: The system shall create detailed logs that include tracing information across microservices.
- **TR-1.7**: The system shall provide detailed metrics for individual microservices via a OpenTelemetry monitoring API.

### Resilience

The system should be resilient to avoid outages.

- **TR-1.8**: Each microservice must continue to operate, even when dependencies have outages. A degraded performance is acceptable, but answering requests, possibly with reduced functionality, warnings, or errors, is a must.

### Configuration

The system should be configured via a configuration file.

- **TR-1.9**: The system exposes exactly one central configuration file, independent of the number of microservices and their technology stacks.
- **TR-1.10**: The system configuration steers the system orchestration, each microservice, and each database.

## 2. Design

The system design should have the following properties:

- **TR-2.1**: All databases support scale out scenarios (i.e., support serving a single database via multiple servers).
- **TR-2.2**: The system supports distributed transactions, ensuring the data integrity across multiple microservices and databases.
- **TR-2.3**: Microservices must communicate with each other only via the microservices' documented APIs.
